# 延迟预算

## 总目标

端到端延迟 (图像采集 → 控制命令输出) < **100 ms**。

## 各模块预算与实测

| 模块 | 预算 (ms) | 当前实测 Mean (ms) | 当前实测 P95 (ms) | 状态 |
|------|----------|-------------------|-------------------|------|
| 预处理 (曝光/3DNR/去雾/CLAHE) | < 5 | 3.5 | 3.7 | ✅ |
| HBD-Net-RT 模型推理 | < 15 | 28.2 | 29.8 | ⚠️ Placeholder 网络未优化 |
| 后处理 (NMS/mask/置信度) | < 5 | 0.2 | 0.2 | ✅ |
| BEV 栅格 (占用+风险) | < 5 | 1.7 | 1.8 | ✅ |
| DWA 路径规划 | < 5 | 2.1 | 2.2 | ✅ |
| 安全状态机 | < 2 | 0.02 | 0.02 | ✅ |
| 其他 (采集/通信等) | < 63 | - | - | 待实测 |
| **总计** | **< 100** | **~36** | **~37** | ✅ 当前达标 |

## 说明

- 模型推理是当前最大瓶颈 (28ms)。Placeholder 网络为随机权重的简单卷积序列，
  未使用 MobileNetV4-small 等轻量 backbone。替换真实模型 + ONNX/TensorRT 后
  预计推理可降至 5-10ms。
- 当前 benchmark 不计入图像采集和通信延迟 (使用随机 tensor 模拟输入)。
  实车部署时需额外统计这部分开销。
- DWA 每次评估 5×9=45 条候选轨迹，向量化后预期可进一步降低至 <1ms。

## 后续优化方向

1. Backbone 替换为 MobileNetV4-small (参数量 <5M) → 推理 <10ms
2. ONNX 导出 + TensorRT FP16 → 推理 <5ms
3. DWA 轨迹评估向量化 → DWA <1ms
4. Python 后处理迁移到 C++ → 后处理 <1ms
